0. Background — GitLab visibility & access model

This section is a primer for readers new to GitLab self-managed. It introduces the concepts the rest of this document uses without further definition: the three visibility levels, the group/project hierarchy, the built-in role ladder, custom roles, and how LDAP/SCIM binds external identity into all of this.

0.1 The three visibility levels (self-managed instance)

Every GitLab project and group has exactly one of these visibility levels. The level controls who can find and read the resource before any role-based access is considered.

Figure 0.1 — The three GitLab visibility levels. "Internal" is the keystone of inner sourcing on a self-managed instance — it lets every authenticated employee discover and read non-confidential code without requiring per-project membership.

Three rules tie these together:

  • A child cannot be more visible than its parent. A Public group cannot contain a Private project, but a Public group can contain Internal or Public projects. This is why the PoC's restricted/ subgroup is Private — anything beneath inherits the Private floor.
  • Visibility filters the audience; roles filter what each member can do. A user who can read a project (because it's Internal) still needs an explicit Developer role to push to it.
  • Visibility is set per-resource, not inherited downward by default — but the rule above means that lowering a parent's visibility cascades a ceiling. (Raising a parent does not raise children.)

0.2 Groups, projects and the membership model

GitLab is organized as a tree. The root of an instance is the top-level group. Below it can be any number of subgroups (which can themselves contain further subgroups), and any group can contain projects (the things that actually hold code).

A user gets access to a project in two ways:

  1. Direct membership — added as a member of that exact project, with a specific role (Guest, Reporter, Developer, Maintainer, Owner).
  2. Inherited membership — a member of any group ancestor of the project. The role inherited is the role they hold on the ancestor, and the highest of (direct, inherited) wins.

In the PoC, we use a third pattern that combines both: group share. Group A can be "shared with" group B at a given role — every member of A then becomes an inherited member of every project under B at that role. This is how LDAP-managed groups (which live under iam-sim/ in the PoC) project access onto the business-domain tree without anyone being a direct project member.

0.3 The built-in role ladder

GitLab ships with seven default roles, ordered by privilege. Higher roles include everything the lower roles can do.

Role Numeric What it grants (project / group level)
Minimal Access5Only meaningful at the top group of a SAML/SCIM-bound instance — the user appears as a member but has no rights to anything inside. Used in the PoC for Dan, who needs to discover Internal projects via Explore but holds no role on them.
Guest10Browse the project (code, issues, MRs, wiki); create issues; comment.
Planner15Guest + manage issues, epics, milestones, iterations.
Reporter20Pull/clone code; read pipeline logs; download artifacts; read non-protected CI variables.
Developer30Push to non-protected branches; open MRs; trigger pipelines on non-protected refs. Cannot push to protected branches, merge to main, edit settings, or read protected variables.
Maintainer40Merge to protected branches; create release tags; edit CI/CD settings; manage protected branches/tags/environments; add members up to Maintainer; register project runners.
Owner50Delete or transfer projects/groups; manage group settings/runners/variables; add Owners; configure SAML/SCIM links.

0.4 Custom roles (Ultimate / EE-only)

The default ladder is coarse — Maintainer is the only role that can edit CI/CD variables, but you may want a release manager to manage variables without giving them write access to source code. Custom roles solve this: you pick a base role and toggle on additional fine-grained abilities.

The PoC defines three custom roles, all on a Reporter base:

  • Promoter — Reporter + admin_cicd_variables. Can manage release-time CI variables without ever pushing code.
  • Operator — Reporter + admin_protected_environments + admin_cicd_variables + read_runners + admin_runners. SRE who manages prod environments + runners without holding write access to code.
  • Security Manager — Reporter + read_vulnerability + admin_vulnerability + admin_merge_request. Vulnerability waiver + approver of security-sensitive MRs.

Custom roles are defined once at the instance level (admin operation), then attached to a group share via member_role_id — that's how a single LDAP group like IAM_DevOps_domain-a_Promoter becomes a Promoter on a specific project.

0.5 Protections that layer on top of roles

Roles set the floor; protection rules set the ceiling on specific actions. The PoC validates four:

  • Protected branches — restrict push and merge to specific roles or users on a per-branch-pattern basis. PoC config: main = "no-one can push, Maintainer can merge, code-owner approval required".
  • Protected tags — restrict tag creation by pattern. PoC config: v* = "Maintainer only".
  • Protected environments — restrict who can deploy to a named environment, and optionally require an approver. PoC config: prod = "Maintainer-floor allowed deployers + 1 approval required from Bob" (the two-person rule).
  • CODEOWNERS — file in the repo that maps path patterns to required reviewer groups. PoC config: /infra/ requires Security Manager group review.

0.6 LDAP / IAM binding (simulated in this PoC)

On a real self-managed instance, identity and group membership come from LDAP/SAML/SCIM, not from GitLab itself. GitLab's job is to enforce access; the lifecycle (joiner / mover / leaver) is owned externally. The real enterprise uses three external identity systems — SSCAM, SailPoint, and P2P — each issuing LDAP groups under different naming conventions.

Wiring a live LDAP/SSO bind into a PoC is out of scope. So the PoC simulates each IAM system as a GitLab group, and each LDAP group as a GitLab subgroup underneath it. Users become members of the subgroup; the subgroup is then shared with the target group/project — which is exactly what a real LDAP-backed GitLab does internally.

The container hierarchy

Under the top-level acme-poc group there is a real side (the business-domain tree) and a simulation side (the iam-sim/ Private subgroup): 

acme-poc/                              ← top-level group (Internal)
├── domain-a/                          ← REAL side — business domain
│   ├── proj-1, proj-2, proj-3
│   └── restricted/restricted-proj-1   ← Private subgroup
├── domain-b/                          ← REAL side — sibling domain
│   └── proj-1, proj-2, proj-3
├── platform/ci-templates              ← REAL side — shared CI
│
└── iam-sim/                           ← SIMULATED IAM side (Private)
    ├── sscam/                         ← simulates SSCAM (project-level LDAP)
    │   ├── domain-a-proj-1_w / _r
    │   └── domain-a-proj-2_w / _r
    ├── sailpoint/                     ← simulates SailPoint (group-level LDAP)
    │   ├── gl-domain-a-{read,dev,maint,owner}
    │   ├── gl-domain-b-{read,dev,maint,owner}
    │   └── gl-restricted-read
    └── p2p/                           ← simulates P2P DevOps roster
        ├── IAM_DevOps_domain-a_Promoter
        ├── IAM_DevOps_domain-a_Operator
        ├── IAM_DevOps_domain-a_SecurityManager
        ├── IAM_DevOps_domain-a_Maintainer
        ├── IAM_DevOps_domain-a_Developer
        ├── IAM_DevOps_domain-a_Reporter
        └── IAM_DevOps_Owner

The three sub-containers (sscam, sailpoint, p2p) exist only to group the simulated LDAP groups the same way the real systems do — they carry no access of their own. They are defined in config.py and created by phase_03_iam_groups.py. Naming mirrors the real LDAP groups one-for-one:

  • <app>_w / <app>_r → SSCAM (Bitbucket-era, project-level — Approach 1)
  • gl-<domain>-{read,dev,maint,owner} → SailPoint (GitLab-native, group-level — Approach 2)
  • IAM_DevOps_<group>_<Role> → P2P DevOps (custom roles — Promoter, Operator, Security Manager)

The two-hop assignment model

Assigning a user to a project is deliberately never direct. It's always two hops:

Hop 1 — put the user in an IAM-sim subgroup (simulates LDAP membership). Phase 4 walks the IAM_MEMBERSHIPS map in config.py and adds each user to one or more IAM-sim subgroups via POST /groups/:id/members. Example: 

IAM_MEMBERSHIPS = {
    "iam-sim/sscam/domain-a-proj-1_w":            [alice],   # Approach 1
    "iam-sim/sailpoint/gl-domain-a-dev":          [alice],   # Approach 2
    "iam-sim/p2p/IAM_DevOps_domain-a_Maintainer": [bob],     # Custom-role roster
    ...
}

At this point Alice has no access to any project yet — she's just listed inside the simulated LDAP group. This matches real life: being in an LDAP group alone does nothing until something binds that group to a resource.

Hop 2 — share the IAM-sim subgroup with a target group/project (the bind). Phase 5 (Approach 1) and Phase 6 (Approach 2) invoke GitLab's "share group with group/project" API:

  • Project share: POST /projects/:id/share with group_id and group_access
  • Group share: POST /groups/:id/share with group_id and group_access

Approach 1 (Hybrid) shares for the PoC:

domain-a-proj-1_w   --share as Developer  -->  domain-a/proj-1
gl-domain-a-dev     --share as Developer  -->  domain-a (inherited to all children)
gl-restricted-read  --share as Reporter   -->  domain-a/restricted

Approach 2 (Target — pure group-level) shares:

gl-domain-b-read    --share as Reporter   -->  domain-b
gl-domain-b-dev     --share as Developer  -->  domain-b
gl-domain-b-maint   --share as Maintainer -->  domain-b
gl-domain-b-owner   --share as Owner      -->  domain-b

Now Alice resolves as: "member of gl-domain-a-dev → that group is shared with domain-a as Developer → therefore Alice is Developer on every project under domain-a, by inheritance." Exactly the behavior a real LDAP-backed GitLab would produce when SailPoint provisions her into gl-domain-a-dev.

Custom roles — the third binding

Phase 1 creates the three custom roles at the instance level. Phase 9 then shares the iam-sim/p2p/IAM_DevOps_domain-a_<Role> subgroups with domain-a/proj-1, passing the member_role_id parameter on the share call. That single call binds the IAM-sim roster to a project and stamps everyone in it with the custom role.

The two deliberate exceptions

Two grants do not route through the IAM-sim tree, on purpose:

  • Dan — added directly to acme-poc top group as Minimal Access (level 5). No IAM-sim membership anywhere. This is how the PoC proves inner-source discovery: any authenticated user can discover Internal projects without being in any LDAP group.
  • Carol on prod — listed by username directly on the prod protected environment's deploy allow-list. Demonstrates user-level exception on top of group-level access, which is sometimes unavoidable for deployment gating.

Every other grant goes through an IAM-sim subgroup — no direct user-to-project assignments.

Why this simulation is faithful

The effective access is identical to a real LDAP bind because in both cases GitLab sees:

"this user is a member of group X, and group X is shared with project/group Y at role Z."

The only difference is how the user lands in group X — manually via the API in the PoC, or via LDAP sync / SCIM / SailPoint provisioning in production. Everything downstream (inheritance, branch protection, CODEOWNERS, audit events, MR approvals) behaves the same way, which is why the PoC validates the design without a live directory.

How to do the share-a-group-with-a-group hop in the GitLab UI
  1. Navigate to the target group (the one that will receive access). Example: acme-poc/domain-b
  2. Left sidebar: Manage → Members.
  3. Click the Invite a group tab (next to Invite members).
  4. In the Select a group to invite dropdown, pick the source IAM-sim group. Example: acme-poc/iam-sim/sailpoint/gl-domain-b-dev
  5. In Select max role, pick the ceiling role (Guest / Reporter / Developer / Maintainer / Owner — or a custom role if one is enabled). This caps access: a user who is Owner in the source group becomes only Developer on the target if "Developer" is selected.
  6. Optional: set an Access expiration date.
  7. Click Invite.
Same flow on a project: Project → Manage → Members → Invite a group. This is the UI equivalent of the POST /groups/:id/share and POST /projects/:id/share calls Phase 5 and Phase 6 make programmatically.

After the invite, the target group's Members → Groups sub-tab shows the share. Individual members of the source group then appear on the target's Members tab labelled "via shared group" — exactly the same label GitLab uses in a production LDAP-backed deployment.

0.7 Putting it together — the access path

When a user attempts an action on a project, GitLab checks:

  1. Visibility — can this user even see this resource? If Private and the user is not a member (direct or inherited): 404.
  2. Effective role — what is the highest role this user holds on this project, considering direct membership, group inheritance, and group shares?
  3. Role-level permission — does that role allow the action? (E.g. push to a non-protected branch needs Developer+.)
  4. Custom-role abilities — if a custom role is in play, does the ability set permit it?
  5. Protection rule — if the action is on a protected branch / tag / environment, does the protection rule allow this user (or their role)?
  6. Two-person rule (if applicable) — for environments with required approvals, is the action approved by a different user?

Every PoC scenario walks one of these rules to its negative or positive outcome. With this background in place, the rest of this document is about which rule each scenario validates and the screenshot proof.

1. Executive summary

Migrating from Bitbucket to GitLab introduces a more powerful but fundamentally different access model. The PoC validates that GitLab's group-centric authorization, custom roles, branch/tag/environment protection, and inner sourcing model can replace the existing Bitbucket-style _w/_r repository-level group pattern without losing the protections we have today and while unlocking patterns that are awkward or impossible in Bitbucket.

The PoC is scaffolded around two parallel authorization approaches:

Figure 1 — Approach 1 (hybrid, transition) vs Approach 2 (domain-centric, target). The PoC validates that both work and that migration between them is non-disruptive.

Result: the PoC ran 13 phases end-to-end against a live GitLab Ultimate instance and 21 UI evidence scenarios. 24 of 24 live verification checks passed. Every claim made in the reference design is supported by an API trace and a browser screenshot.

2. What the PoC proves

Five top-level claims, each backed by one or more scenarios.

Claim 1 — Inheritance flows down, siblings are isolated

Granting a role at a group level propagates to every subgroup and project beneath it. Sibling groups (different domains) cannot see each other unless explicitly shared. Access can be raised at a lower level, but never lowered below the inherited floor.

Validated by: Scenarios 02 (user-membership), 08 (inheritance), 08b (sibling-isolation), 10h (variable inheritance).

Claim 2 — Branch / tag / environment protection enforces the right boundaries at the right people

Protected branches block direct push and require code-owner approval. Protected tags restrict release tag creation to Maintainers. Protected environments combine an allowed-deployer list with a separate approval-required step (two-person rule for production).

Validated by: Scenarios 04 (protected-branch), 05 (protected-env), 06 (protected-tags), 10b (manual-jobs).

Claim 3 — Custom roles deliver fine-grained authority without Maintainer creep

Three custom roles — Promoter (manages release-time CI variables), Operator (manages prod environment + release branches), Security Manager (waives vulnerabilities, approves security-sensitive MRs) — let role-specific personas do their job at Reporter base level + a targeted ability set, instead of needing Maintainer-grade write access they don't otherwise need.

Validated by: Phase 1 (custom-role creation), Phase 9 (custom-role assignment), Scenarios 10b (manual-jobs), 10f (cicd-settings) for negative/positive checks.

Claim 4 — Inner sourcing supports forked contributions without granting upstream write access

External contributors with only Minimal Access at the top group can discover Internal projects, fork them into their personal namespace, open MRs back to the upstream, and have those MRs merged after a Trusted-Committer review. They never hold push or merge rights on the upstream project.

Validated by: Scenario 07 (inner-source), 10c (fork-isolation), 10c2 (fork-as-dan).

Claim 5 — Approach 1 → Approach 2 migration is non-disruptive

Removing the Bitbucket-style project-level shares (Approach 1) does not break access for users who are already members of the corresponding SailPoint domain group (Approach 2). The migration can be done one project at a time, with no observable outage.

Validated by: Phase 13 (migration), Scenario 09 (audit).

3. The PoC layout

The hierarchy is intentionally flat — business domains (long-lived application groupings) sit directly under the top-level group. Each app appears in exactly one place in the tree, regardless of how many environments it deploys to.

Figure 2 — The PoC group hierarchy. Each business domain sits directly under the top group. The iam-sim/ subtree simulates the SSCAM, SailPoint and P2P IAM systems that hold real LDAP groups in production. platform/ hosts shared CI templates.
What about deployment zones (live-prod, non-live, cloud landing zone)? Deployment-model isolation is not expressed by group nesting in this hierarchy. It is expressed instead by:
  • Environment names on protected environments (staging, prod) — see Scenario 05.
  • Environment-scoped CI variables (e.g. PROD_DEPLOY_TOKEN with environment_scope=prod) — only visible to jobs that declare the matching environment. See Scenarios 10a, 10i.
  • Group-level runners tagged by zone (e.g. a prod-runner with the Protected flag) — only picks up jobs from protected refs, never from feature branches. See Scenario 10e.
This keeps each app in exactly one place in the tree (avoiding the per-zone duplication that would otherwise force triple membership and triple CI variable management) while still giving you per-zone runner isolation, per-zone secrets, and per-zone audit reporting.
