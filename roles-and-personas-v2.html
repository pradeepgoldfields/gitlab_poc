The "effective role(s)" column shows what the design intends. On a self-managed instance without LDAP sync, GitLab's runtime permission resolver may not honour the full org-driven chain (see poc-overview-v2.html section 7 for the full explanation). The capability matrix below assumes a working LDAP-bound production environment, which is the design's deployment target.

Footnotes
  1. Even Owners and Maintainers can't push directly to main — protected branch rule is "Allowed to push: No one; Allowed to merge: Maintainers". Merging an MR works.
  2. Pat / Tina / Bob aren't the prod-env allowed deployer. Carol is the only individually-listed deployer.
  3. Sam approves MRs via the admin_merge_request ability — designed for security-sensitive MRs (e.g. /infra/ per CODEOWNERS) without granting write access to source code.
  4. Maintainer can add members up to Maintainer level. Adding an Owner requires an Owner.
  5. Carol plays deploy_prod, but the deployment then transitions to "waiting for approval" because the prod environment requires Bob's approval (two-person rule).
  6. GitLab blocks self-approval on a deployment Carol initiated — even if she were on the approver list.
  7. Operator's admin_protected_environments ability covers environments. Carol does NOT have admin_protected_branches, so branch rules remain Maintainer-only.
  8. Ed's access is project-scoped to proj-3 only. He can clone / push to feature branches / open MRs / read CI variables on proj-3, but on proj-1 and proj-2 he can only read (Internal visibility).
  9. Fred is a Developer on domain-b/subdomain-b, NOT on domain-a/subdomain-a. He can read proj-1 (Internal) but cannot push, MR, or run pipelines on it. His write access is on proj-4 / proj-5 / proj-6.
  10. Dan opens MRs from a fork (v2-dan/proj-1), not the upstream. He can't create a feature branch on upstream proj-1.
