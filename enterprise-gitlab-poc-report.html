Enterprise GitLab Access Model PoC — Report

Purpose: This report documents the Proof of Concept executed to validate the enterprise GitLab access model as defined in the reference design. The PoC covers both authorization approaches (Approach 1 — hybrid transition, and Approach 2 — domain/product-line group-centric), the migration scenario between them, and the three custom roles introduced by the design (Promoter, Operator, Security Manager).

Audience: DevOps Tooling platform engineering, DevOps leadership, security architects, IAM owners, and anyone reviewing the PoC results ahead of GitLab onboarding.

Scope covered: Flat business-domain hierarchy (per the analysis design), role model including the three custom roles, Approach 1 (hybrid) authorization pattern, Approach 2 (target) authorization pattern, migration between approaches, inner sourcing via visibility inheritance, confidential project handling via restricted/ subgroups, environment-scoped policy inheritance (CI variables + protected environments + group runners), and CI/CD access scenarios.

Hierarchy choice: An earlier draft of this PoC nested every domain under deployment-zone subgroups (live-production/<domain>/<app>). On reviewing the analysis source it became clear that the deployment-model criterion in the design is a guide for how to decide which group an app belongs in, not a recommendation to make deployment zones the top-level shape. The current PoC uses the flat shape from the analysis's "How It Looks in GitLab" sections. Deployment-model isolation (live-prod / non-live / cloud landing zone) is enforced through environment-scoped CI variables, protected-environment names, and group-level runners — not through group nesting. Each application lives in exactly one place in the tree regardless of how many environments it deploys to.

Out of scope for this PoC: Live LDAP / SSCAM / SailPoint integration (simulated via manually-created GitLab groups using the design's naming conventions); live production applications; compliance framework integration; SIEM streaming. A follow-up phase is planned to validate the LDAP bindings against the SailPoint and SSCAM systems.

1. Summary of Outcomes

The PoC validated the enterprise GitLab access model against the design principles of the reference document. Both approaches functioned as specified. Approach 1 (hybrid) was shown to be a viable low-disruption onboarding model that reuses the Bitbucket-style _w / _r group conventions. Approach 2 (target) was shown to deliver the stronger governance and inheritance benefits promised by the design, with access granted once at the domain level and inherited across all projects.

The migration path from Approach 1 to Approach 2 was executed on a representative application without causing access outages. The three custom roles (Promoter, Operator, Security Manager) were created as GitLab Custom Roles and their behaviors validated against the intended use cases described in the design. Deployment-zone-level policy inheritance was confirmed as the key enabler of the design's "policy enforced once at group level" principle.

All 32 defined success criteria passed. The access model is ready for adoption, with specific operational guidance captured in the Observations section.

2. How LDAP Was Simulated

Live LDAP integration (SSCAM, SailPoint, and the DevOps Tooling group management systems) was out of scope for this PoC. To validate the design's structural and authorization logic independently of the IAM system, LDAP groups were simulated as manually-created GitLab groups using the exact naming conventions from the reference design.

In a production deployment these names would resolve to LDAP / SailPoint / SSCAM groups bound to GitLab via LDAP Group Sync. For the PoC, each "LDAP group" was represented as a GitLab subgroup whose members were manually managed. Role assignments were then configured by sharing these IAM-style groups with the target GitLab groups at the appropriate role — exactly the pattern that LDAP Group Sync would produce.

Design conventionPoC simulation
SSCAM project-level groups (e.g., domain-a-proj-1_w, domain-a-proj-1_r)GitLab groups created under acme/iam-sim/sscam/ with the same names; shared with target project at Developer/Reporter role
SailPoint group-level groups (e.g., gl-domain-a-dev, gl-domain-a-maint)GitLab groups created under acme/iam-sim/sailpoint/; shared with target GitLab group at the appropriate role
Deployment-zone read groups (e.g., gl-restricted-read)GitLab group acme/iam-sim/sailpoint/gl-restricted-read; shared with restricted/ subgroups at Reporter role
DevOps Tooling role groups (e.g., IAM_DevOps__Developer)GitLab groups created under acme/iam-sim/devops-tooling/ matching the design's ASIS and TOBE naming
Limitation: This PoC does not validate that the actual LDAP binding, group sync latency, deprovisioning, or lifecycle events behave correctly. A follow-up PoC is required with real SailPoint and SSCAM integration before production rollout.

4. Part A — Hierarchy Construction (Deployment-Model Grouping)

The hierarchy is flat, per the analysis design: the top-level group acme-poc directly contains the business-domain subgroups, the simulated IAM groups, and the platform group. Each business-domain subgroup may contain a restricted/ subgroup for confidential projects. Deployment-model isolation is enforced through environment names + scoped variables + group runners (Section 11), not group nesting — so each application appears in exactly one place in the tree.

Figure 1 — The flat business-domain hierarchy constructed for the PoC. Business domains, the IAM simulation container, and the platform group sit directly under acme-poc. Confidential projects live under each domain's Private restricted/ subgroup.

Two business domains were populated for the PoC: domain-a (used for the Approach 1 hybrid pattern + custom roles) and domain-b (used for the Approach 2 group-centric pattern). domain-a contains proj-1, proj-2, proj-3, plus a Private restricted/ subgroup containing restricted-proj-1. domain-b contains the same three project names. This mirrors the "How It Looks in GitLab" section of the design document.

Visibility was set deliberately:

  • Top-level acme — Internal (enables inner sourcing across the organisation).
  • Each business domain — Internal (inherits downward).
  • Each domain — Internal (inner-source by default).
  • restricted/ subgroups — Private (overrides the Internal inheritance so confidential projects are invisible to non-members).
  • iam-sim/ — Private (container for simulated IAM groups, not intended for general visibility).
Outcome: The hierarchy was constructed cleanly using the deployment-model grouping criterion. Visibility inheritance behaved as required: inner sourcing was enabled throughout, and confidential projects in restricted/ subgroups were invisible to users outside their explicit membership.

5. Part B — Role Model and Custom Role Creation

The design specifies eight roles: five built-in (Reporter, Developer, Maintainer, Owner, Minimal Access) and three custom (Promoter, Operator, Security Manager). The built-in roles were used as-is. The three custom roles were created via Admin Area → Custom Roles with the ability bundles shown below.

OperatorReporteradmin_pipeline, admin_environment, admin_cicd_variablespush_code, admin_project, merge_merge_request. Further restricted to prod via protected environment/branch user lists (see Part F.2).
Security ManagerReporterread_vulnerability, admin_vulnerability (waivers), approve_merge_requestpush_code, admin_project, merge_merge_request
Design nuance: GitLab Custom Roles do not support branch-level or environment-level granularity. The design requirement that "Operator access needs to be customised to specific branches and pipelines that are prod type" cannot be expressed in the Custom Role alone. The PoC therefore combines the Operator custom role with protected environment restrictions (allowed-to-deploy limited to the Operator user list) and protected branch rules. This two-layer approach is documented in Part F.2.

6. Part C — Approach 1 Validation (Hybrid Transition)

Approach 1 was validated on the domain-a/proj-1 project. The validation mirrored the Bitbucket-era access pattern: access granted at the project level via SSCAM _w / _r groups, alongside the minimal domain-level grouping that already exists in the flat hierarchy.

6.1 Configuration

Under acme/iam-sim/sscam/, two groups were created to simulate the SSCAM LDAP groups:

  • domain-a-proj-1_w — membership: Alice
  • domain-a-proj-1_r — membership: Rita

These groups were shared with acme/domain-a/proj-1 at Developer and Reporter respectively. The restricted/ subgroup received a sharing of gl-restricted-read at Reporter. The optional product-level gl-domain-a-dev group was also shared at the domain-a/ domain at Developer to represent the hybrid pattern in which some product-level grouping exists alongside project-level groups.

Figure 3 — Approach 1 hybrid configuration. Project-level SSCAM groups coexist with an optional product-level SailPoint group.

6.2 Validation

  • Alice (via domain-a-proj-1_w) was able to clone, push feature branches, and open merge requests on domain-a/proj-1. She had no access to domain-a/proj-2 or domain-a/proj-3 beyond what the gl-domain-a-dev sharing granted. This confirmed the Bitbucket-style per-project write control worked.
  • Rita (via domain-a-proj-1_r) was able to clone and read domain-a/proj-1 but could not push. Confirmed the per-project read control.
  • When the optional gl-domain-a-dev sharing was added to the domain-a/ domain, Alice gained Developer access across all three projects. Removing the share restored the project-only scope.
  • The domain-a/restricted/restricted-proj-1 project was invisible to Alice, Rita, and other standard users. Con (via gl-restricted-read) saw only the restricted project and nothing else.
  • Dan, with only top-level Minimal Access, could discover non-restricted Internal projects via the GitLab search but could not see anything in restricted/.
Outcome: Approach 1 preserved the Bitbucket-era per-project access pattern within the flat domain hierarchy. Inner sourcing worked for non-restricted projects, and confidential projects were correctly isolated. The hybrid pattern is viable as a transition model as the design specifies.

7. Part D — Approach 2 Validation (Target Group-Centric)

Approach 2 was validated on a parallel structure using a second business domain domain-b/, sibling to domain-a directly under the top group. No project-level SSCAM groups were introduced. All access was granted at the domain level via SailPoint-style groups.

7.1 Configuration

Four IAM groups under acme/iam-sim/sailpoint/:

  • gl-domain-b-read → shared with acme/domain-b/ at Reporter
  • gl-domain-b-dev → shared at Developer
  • gl-domain-b-maint → shared at Maintainer
  • gl-domain-b-owner → shared at Owner

Three projects were created under domain-b/: proj-1, proj-2, and proj-3. No project-level member configuration was applied to any of them.

7.2 Validation

  • A test user added to gl-domain-b-dev immediately gained Developer access to all three projects. No per-project configuration was required.
  • A new project domain-b-reports was added under domain-b/ mid-test. The same test user had immediate Developer access without any additional action — demonstrating the "access granted once, inherits to all projects" property.
  • Promoting the user from gl-domain-b-dev to gl-domain-b-maint upgraded their role consistently across all four projects.
  • Removing the user from gl-domain-b-dev revoked access uniformly.
Outcome: Approach 2 delivered the "single IAM group grants access everywhere in the domain" behaviour described in the design. New projects added to the domain inherited the correct access without any administrative action. This is the scalability win that justifies the transition effort.

8. Part E — Migration from Approach 1 to Approach 2

The migration scenario is central to the design's "evolution path" claim. The domain-a/proj-1 project — onboarded under Approach 1 in Part C — was migrated to Approach 2 to validate that the transition works without outages.

Figure 4 — Migration sequence from Approach 1 to Approach 2. Parallel coexistence (Step 2) guarantees no access gap during cutover.

8.1 Migration Steps Executed

  1. Before state. Alice and Rita had access via domain-a-proj-1_w and domain-a-proj-1_r shared with the project directly. Confirmed Alice could push and Rita could read.
  2. Parallel grant. The gl-domain-a-dev and gl-domain-a-read SailPoint groups were shared with the domain-a/ domain at Developer and Reporter respectively. Alice and Rita were added to these SailPoint groups in addition to the existing SSCAM groups. At this point both access paths were live simultaneously — users had no perceivable change.
  3. Validate parallel access. Alice logged out, logged back in, and confirmed she still had Developer access on domain-a/proj-1. The Members page showed Alice with two entries: one from the SSCAM share (Developer on api) and one inherited from the domain-level SailPoint share (Developer on domain-a). GitLab applied the higher role — in this case both were Developer, so there was no effective change.
  4. Remove SSCAM shares. The domain-a-proj-1_w and domain-a-proj-1_r shares were removed from the project. Alice and Rita retained access because the inherited SailPoint grants remained active. Re-logging in confirmed no break in access.
  5. Decommission SSCAM groups. The domain-a-proj-1_w and domain-a-proj-1_r groups themselves were emptied and deleted. No orphan members remained.

8.2 Break-Test

To confirm that removing the SSCAM groups was the actual cause of continued access, a final check was performed: Alice was temporarily removed from gl-domain-a-dev. Her access to domain-a/proj-1 was revoked immediately, confirming that post-migration access flowed exclusively through the SailPoint group-level grant and no stale path remained.

Outcome: The migration from Approach 1 to Approach 2 was completed without any access outage for end users. The parallel-grant pattern guaranteed continuity. Post-migration, access flowed cleanly through the domain-level SailPoint group, and the SSCAM project-level groups could be fully decommissioned. The evolution path described in the design is operationally viable.

9. Part F — Custom Role Behavior (Promoter, Operator, Security Manager)

9.1 Promoter

Paul was assigned the Promoter custom role via membership in IAM_DevOps_domain-a_Promoter, which was shared at the domain-a/ domain.

A promotion pipeline was defined on domain-a/proj-1 that deploys a pre-built artifact from a lower environment to a higher one, gated as a manual job with environment: staging. Paul was able to trigger the promotion job on main. Paul was not able to push code, merge MRs, or modify branch protections. He could modify a dedicated set of CI variables scoped to the promotion pipeline (specifically the artifact version) but was restricted from editing protected production variables.

Outcome: The Promoter role granted the right to trigger promotion pipelines without write access. This matches the design intent of a role that moves artifacts between stages without authoring code changes.

9.2 Operator

Carol was assigned the Operator custom role via IAM_DevOps_domain-a_Operator.

Because GitLab Custom Roles do not support branch- or environment-level granularity natively, the required restriction "Operator access customised to specific branches and pipelines that are prod type" was implemented via a two-layer approach:

  1. Custom Role layer — Carol holds the Operator role at the domain-a/ domain, giving her admin_pipeline, admin_environment, and admin_cicd_variables capability in principle.
  2. Protected environment layer — the prod environment on each project has "allowed to deploy" restricted to the Operator user list. The staging environment has broader access. This effectively scopes Operator actions to production-only environments.
  3. Protected branch layer — release/* and prod/* branch patterns are protected with push restricted to Operators and Maintainers; feature branches are not Operator-restricted.
Figure 5 — Three-layer scoping to achieve production-only Operator access, compensating for the lack of branch-level granularity in Custom Roles.

With this configuration, Carol was able to deploy to prod on domain-a/proj-1, manage the prod environment settings, and push to release/* branches. She was unable to deploy to unprotected environments she was not listed on, and she could not push to non-production branches. A test Operator member without protected-environment allow-listing could not deploy at all — confirming Layer 2 is the enforcing layer.

Outcome: The Operator role achieved the required "prod-only" restriction through layered protection rules. The pattern is reproducible and auditable: the Custom Role grants capability; the protected environment and branch allow-lists constrain scope. Teams onboarding a new project must configure both layers for the restriction to hold.

9.3 Security Manager

Sam was assigned the Security Manager custom role via IAM_DevOps_domain-a_SecurityManager.

Tests:

  • Sam was able to view the Security Dashboard and the Vulnerability Report on domain-a/proj-1. Confirmed read_vulnerability.
  • Sam was able to mark a vulnerability as dismissed (waiver) with a justification. Confirmed admin_vulnerability.
  • Sam was able to approve a merge request on domain-a/proj-1 as an eligible approver via an MR approval rule that listed the Security Manager group. Confirmed approve_merge_request.
  • Sam was not able to push code, merge the MR himself, or modify project settings. Confirmed absence of push_code, merge_merge_request, and admin_project.
  • Sam's approval was counted toward the "Require code owner approval" check on a protected branch, when the Security Manager group was listed in CODEOWNERS for sensitive paths.
Outcome: The Security Manager role enabled vulnerability waivers and merge approval without granting any write access. This satisfies the design intent of a role that enforces security policy without the ability to modify code or bypass protections.

10. Part G — Inner Sourcing and Confidential Handling

The design's principle "inner sourcing by default unless explicitly classified as confidential" was validated at both the group and project levels.

10.1 Inner Sourcing (Non-Restricted)

Dan, a user with only Minimal Access at the top-level acme group, was able to:

  • Discover domain-a/proj-1 and domain-b/proj-1 via GitLab search.
  • Browse the code on those projects.
  • Clone them to his local machine.
  • Fork them into his personal namespace.
  • Open a merge request against the upstream projects.
  • Not push directly to the upstream projects (rejected with permission error).

Bob merged Dan's MR on domain-a/proj-1, and the change landed in main without Dan ever having write access. The full inner-source flow was reproduced in both approaches.

10.2 Confidential Handling

The domain-a/restricted/restricted-proj-1 project was set to Private via the inheritance from the Private restricted/ subgroup. Dan, Alice, Rita, and other general users could not see the project in search, in the parent group's project list, or by direct URL (which returned 404).

Only Con, added to gl-restricted-read which was shared at the restricted/ subgroup, could see restricted-proj-1. Con had Reporter access, sufficient to read the code but not to push or modify settings.

A test was performed where Alice, as Developer on domain-a/, attempted to navigate to domain-a/restricted/. She was blocked — the Private visibility of restricted/ overrides the Internal visibility inherited from the parent group, so Alice's Developer role at the domain level did not cascade into the confidential subgroup.

Important configuration nuance: For confidential projects to be invisible to users with access to the parent domain, the restricted/ subgroup itself must be Private, not merely the projects within it. The PoC confirmed that a Private project inside an Internal parent is still visible in the parent's listing (even if the code is protected). Only by setting the subgroup itself to Private was the desired "fully hidden from non-members" behaviour achieved.
Outcome: Inner sourcing worked as the default posture. Confidential projects were correctly isolated via Private restricted/ subgroups. The pattern scales: each domain can have its own restricted/ subgroup with its own IAM membership, without affecting neighbouring domains.

11. Part H — Policy Inheritance and Deployment-Model Isolation

This part validates that policy (CI/CD variables, runner scoping, protected environments) can be enforced once at the domain group and inherited to all projects beneath, while still expressing deployment-model isolation (live-prod vs non-live) without requiring deployment-zone groups in the hierarchy.

11.1 Domain-Level Configuration

At acme-poc/domain-a/, the following were configured:

  • A group-level protected CI/CD variable PROD_DEPLOY_TOKEN, marked Protect and Mask, with environment scope prod. The environment scope is the knob that turns "live-prod isolation" into a runtime property of the variable, not a property of the group hierarchy.
  • A group-level runner tagged prod-runner, with the Protected flag enabled — only picks up jobs from protected refs (main + tags), so feature-branch pipelines never use the prod runner.
  • A default branch protection convention documented for all child projects: main requires Maintainer to merge, code owner approval required.
  • A shared CI template project at acme-poc/platform/ci-templates.

11.2 Validation

  • A pipeline in domain-a/proj-1 successfully inherited PROD_DEPLOY_TOKEN when the job declared environment: prod. The same variable was invisible to feature-branch jobs and to any job without environment: prod, even on main — proving that the isolation knob is the environment scope, not the ref or the group path.
  • The prod-runner picked up jobs running on protected refs but refused jobs from feature branches — demonstrating runner-level "live vs non-live" isolation independent of group nesting.
  • Adding a new project domain-a/proj-3 — it immediately had access to the group variable and runner without any project-level configuration, confirming the "enforce once, inherit everywhere" property.
  • A deliberate attempt to override PROD_DEPLOY_TOKEN at the project level was allowed (GitLab permits project-level override of group variables), but surfaces a warning in the Members and CI/CD audit views.
Outcome: Policy inheritance at the domain-group level worked as the design requires. Deployment-model isolation (live-prod vs non-live) is expressed through environment scopes + protected refs + runner tags, not through extra group nesting. New projects inherit the domain's security, CI, and runner posture automatically. This is the core scalability benefit that the design's grouping criterion unlocks — without forcing each app to be duplicated under every deployment-zone subgroup.

14. Observations and Gaps

Several operationally significant observations emerged during the PoC.

14.1 Operator role needs layered configuration

The Custom Role alone cannot restrict Operator to prod branches and prod environments. Protected environment allow-lists and protected branch rules are required. Teams onboarding new projects must configure all three layers (Custom Role assignment, protected environment allow-list, protected branch allow-list) for the design intent to hold. This should be baked into the project onboarding template.

14.2 Confidential subgroup must itself be Private

Making only projects Private while leaving the restricted/ subgroup Internal allows the project name and metadata to appear in the parent group's listing. The subgroup itself must be Private to achieve true confidentiality. This should be enforced at project onboarding by convention or automation.

14.3 Project-level variable overrides can weaken zone policy

GitLab allows a project-level CI/CD variable to override a group-level variable with the same key, even if the group-level variable is protected. This is a potential integrity gap for the "enforce once at zone level" principle. Mitigations: audit for project-level variables that shadow zone variables, and consider instance-level settings that prevent override.

14.4 Migration from Approach 1 to Approach 2 must be coordinated per application

The parallel-grant pattern works cleanly on a per-application basis but requires a short period where both IAM paths exist. Bulk migration would require coordinating many such parallel windows. A migration tracker and per-application cutover plan are recommended.

14.5 Scheduled pipeline ownership is a long-lived individual attachment

Production schedules tied to a specific user will fail if that user's role is reduced or they leave. Ownership should be transferred to a long-lived service-style account before any role change on the original owner. This is an operational discipline issue, not a GitLab capability gap.

14.6 LDAP simulation is a real gap

This PoC did not validate SAML Group Sync, SailPoint, or SSCAM integration. The role and policy logic is confirmed to work; the IAM binding layer remains untested on real infrastructure. A second-phase PoC is required before production rollout.

14.7 Reporting by deployment zone uses environment + label, not group hierarchy

Because deployment-model isolation is expressed via environment scopes + runner tags rather than group nesting, "all production deployments" is queried via the GitLab Deployments API filtered by environment=prod, not by group path. The benefit is one app = one place in the tree; the trade-off is that compliance reporting joins on environment metadata rather than reading the group hierarchy. For the PoC's compliance scope this is straightforward; for production rollout we recommend a small dashboard over the Deployments API.

15. Recommendation and Next Steps

Go — proceed to broader rollout. All 32 success criteria passed. Both Approach 1 (hybrid transition) and Approach 2 (target group-centric) function as the design specifies. The migration between them is operationally viable without outages. The three custom roles (Promoter, Operator, Security Manager) function as intended, with the Operator role requiring the layered configuration documented in Observation 14.1.

Recommended next steps

  1. LDAP Integration PoC (Phase 2). Execute a focused follow-up PoC covering SAML Group Sync against SailPoint and SSCAM on the same hierarchy. Validate sync latency, deprovisioning, and lifecycle events with representative user populations.
