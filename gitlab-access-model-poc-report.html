GitLab Access Model PoC — Report

Purpose: This document describes the Proof of Concept executed to validate the GitLab enterprise access model on a self-managed instance. It records what was built, how access was assigned, and the outcomes of every access and CI/CD scenario tested.

Audience: Platform engineering, DevOps leadership, security architects, and anyone reviewing the PoC results ahead of broader rollout.

Scope covered: Group and subgroup hierarchy, project creation, role assignment, branch and tag protection, environment protection and deploy approvals, inner source contribution flow, access inheritance, CI/CD variables, runner scoping, job tokens, manual jobs, and scheduled pipelines. Out of scope for this PoC: SSO, SCIM, compliance frameworks, SIEM integration.

How to verify each use case in the UI: Each "Outcome" block below is followed by a Verify in UI callout listing the exact GitLab pages a reviewer can open to confirm the result. The URLs use the narrative paths from this report (acme/verticals/domain-a/service-a etc.). Substitute <gitlab-host> with your instance hostname and replace the group/project paths with the equivalents in your environment.

1. Summary of Outcomes

The PoC validated the end-to-end access model against 25 defined success criteria. All scenarios executed as expected. The role hierarchy (Guest → Reporter → Developer → Maintainer → Owner) enforced the intended boundaries on repository actions, and the CI/CD layer (protected variables, protected environments, protected tags, runner scoping, job token allowlists) respected the same access boundaries in pipeline execution.

Key findings:

  • Role-based inheritance behaved consistently — a single assignment at a group propagated to all subgroups and projects beneath.
  • Protected CI variables were genuinely isolated from feature branch pipelines and from fork pipelines, even when external contributors were allowed to run CI.
  • Production deployments could be restricted to specific users and gated by two-person approval.
  • Inner source contribution via fork-and-MR worked without granting write access to external contributors.
  • No additional GitLab configuration was required beyond the built-in access controls — no third-party tooling, compliance framework, or IdP integration was needed for the access model itself to function.

4. Part 1 — Hierarchy Construction

The PoC began by constructing the group and project hierarchy shown in Figure 1. The top-level group acme was created at Private visibility. Two subgroups were added beneath it: platform (Internal visibility, to enable inner source on shared capabilities) and verticals (Private). Within verticals, the domain-a subgroup was created as the pilot vertical. All groups and projects were initialized with default settings. Each project was created with a README to enable immediate clone and push.

Outcome: The hierarchy was constructed cleanly through the web UI. Navigation confirmed that acme contained platform and verticals, that verticals contained domain-a, and that projects appeared in the correct parents.

6. Part 3 — Baseline Access Enforcement

Before any protection rules were applied, the baseline role model was exercised to confirm GitLab's default enforcement.

Alice, as Developer, successfully cloned service-a, created a feature branch, committed, and pushed. Her attempt to access project Settings was rejected with a permission error. The Delete Project option was absent from her view, and a direct URL attempt to the project settings edit page returned 403.

Carol attempted to push to ci-templates (a project on which she held no role). The clone succeeded (because ci-templates is Internal), but the push was rejected with "You are not allowed to push code to this project."

Outcome: Baseline role enforcement worked as documented. Developers could write to projects where they held the role; they were blocked from settings and destructive actions. Users with no role were blocked from writing even to Internal projects.

7. Part 4 — Protected Branch and Merge Flow

The main branch of service-a was protected with the following settings: Allowed to merge — Maintainers; Allowed to push — No one; Allowed to force push — No; Require approval from code owners — On.

Merge request approval rules were configured: prevent approval by author; prevent approvals by users who add commits; remove all approvals when commits are added; minimum approvals required — 1 (later raised to 2 to verify scaling).

A CODEOWNERS file was added to the repository root naming Bob as default reviewer and a security group as required reviewer for /infra/ paths.

Figure 3 — Merge flow with branch protection and approval rules. Red boxes indicate blocked actions; green indicate successful ones.

The flow was exercised by each persona:

  • Alice pushed a feature branch successfully.
  • Alice attempted to push directly to main. The push was rejected with "You are not allowed to push code to protected branches on this project."
  • Alice opened a merge request from her feature branch to main. The MR showed "1 approval required" and listed Bob as an eligible approver.
  • Alice attempted to approve her own MR. The Approve button was disabled.
  • Alice attempted to merge. No merge button was available, because Developer role does not grant merge to protected branches.
  • Bob approved and merged successfully.
  • The approval rule was raised to require 2 approvals. A new MR opened by Alice then remained blocked even after Bob approved, because no second eligible approver existed. This verified the mechanism; the rule was then restored to 1 for subsequent scenarios.
Outcome: Branch protection, self-approval prevention, and approval scaling all behaved as designed. Authors could not approve their own work. Developers could not merge to protected branches. Approval counts could be increased to enforce stricter reviews.
Verify in UI:
  • Protected branches config (main → Maintainers can merge, no one can push): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/repository#js-protected-branches-settings
  • MR approval rules (prevent author approval, min approvals): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/merge_requests
  • Open / merged MRs and approval state: https://<gitlab-host>/acme/verticals/domain-a/service-a/-/merge_requests
  • CODEOWNERS file in repo root: https://<gitlab-host>/acme/verticals/domain-a/service-a/-/blob/main/CODEOWNERS

8. Part 5 — Protected Environment and Deployment

A minimal CI pipeline was added to service-a with three stages: build, deploy to staging, and deploy to prod. The staging deploy was configured to run automatically on main. The prod deploy was configured as a manual job gated by a release tag.

Two environments were protected. Staging allowed Developers and Maintainers to deploy. Prod was restricted to Carol individually (representing the SRE role) with a required approval from Bob (representing a two-person rule).

Figure 4 — Protected environment deployment flow with two-person rule.

The deployment scenario was executed by tagging v0.1 on main, which triggered the prod deploy job in a waiting-for-manual state. Alice attempted to trigger the job and the play button was disabled with a permission tooltip. Carol triggered the job; it transitioned to "waiting for approval". Bob approved via the environment's deployment approval control, and the job completed successfully.

Outcome: Production deployments were successfully gated by user-specific authority and two-person approval. Unauthorized users could not initiate deploys, and even the authorized user could not deploy alone.

10. Part 7 — Inner Source Contribution

Inner source was validated using Dan as the external contributor. Dan held only Guest on the top-level acme and had no role on domain-a or service-a.

To enable discovery, service-a was switched from Private to Internal (along with its parent subgroups, as required by the inheritance rule that a child cannot be more visible than its parent). Dan was then able to find service-a via search, browse it, and fork it into his personal namespace.

Figure 5 — Inner source contribution via fork and MR, with no write access granted.

Dan cloned his fork, committed a small change adding himself to a CONTRIBUTORS.md file, pushed to his fork, and opened a merge request targeting acme/verticals/domain-a/service-a:main. He then attempted to push the same branch directly to the upstream repository. This push was rejected.

Bob reviewed, approved, and merged the MR. Dan's change landed on main without Dan ever having write access to the upstream project.

Outcome: Inner source worked end-to-end. External contributors could discover, fork, and propose changes. Merge authority remained entirely with the host vertical.

11. Part 8 — Inheritance and Sovereignty

To verify inheritance behavior, a second project service-b was created inside domain-a without any direct member assignment. Alice was able to clone and push branches to service-b immediately — access inherited from her Developer role on domain-a.

To verify sibling isolation, a second vertical acme/verticals/identity was created with a project service-x inside. Alice had no access to this sibling vertical despite her role on domain-a. The project was invisible in her navigation and direct URL access returned 404.

To verify access elevation at lower levels, Alice was granted Maintainer specifically on service-a. Her effective role on service-a became Maintainer (the higher role), while she remained Developer on domain-a overall and Developer on service-b.

An attempt to reduce Alice's access on service-a to Guest was rejected by GitLab, which preserved her inherited Developer role from the parent group.

Figure 6 — Inheritance and sibling isolation: access flows down one branch only; the highest assignment wins.
Outcome: Inheritance propagated access downward correctly. Sibling verticals were fully isolated. Access could be raised at lower levels but not lowered below the inherited floor.

13. Part 10 — CI/CD Access Scenarios

A set of nine CI/CD scenarios were executed to validate that the access model governs pipeline behavior as expected. These scenarios extended the pipeline built in Part 5 and tested variables, runners, job tokens, manual jobs, and schedules.

13.1 Scenario A — Protected Variables on Feature vs. Protected Branches

Two CI/CD variables were added to service-a. PROD_SECRET was marked Protect and Mask, with environment scope prod. DEV_CONFIG was left unprotected with wildcard scope.

A debug job was added to the pipeline that echoed both variables on every branch. When Alice pushed to a feature branch, the pipeline ran and the log showed DEV_CONFIG with its value while PROD_SECRET was blank. After merging to main, the same job running on the protected ref still did not expose PROD_SECRET because the job lacked an environment: prod declaration. Only when a job explicitly declared the prod environment did the variable become available.

Figure 7 — Visibility of the protected, environment-scoped variable PROD_SECRET across pipeline contexts.
Outcome: Protected variables required both a protected ref and a matching environment declaration to be exposed. The combination of the Protect flag and the environment scope was the correct configuration for production secrets.
Verify in UI:
  • CI/CD variables (PROD_SECRET protected+masked, env scope prod; DEV_CONFIG unprotected): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/ci_cd#js-cicd-variables-settings
  • Pipeline list (compare debug job logs across feature vs main): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/pipelines

13.2 Scenario B — Manual Job Authorization

The existing deploy_prod manual job targeted the protected prod environment restricted to Carol. When Alice navigated to a pipeline containing this job, the play button was disabled with a "You do not have permission" tooltip. When Carol navigated to the same pipeline, the job transitioned to "waiting for approval" after she played it, and completed after Bob approved.

A second manual job run_adhoc was added without any environment declaration. Alice was able to play this job successfully, confirming that manual jobs without environment protection are gated only by project role — not by any user-specific list.

Outcome: User-specific restriction of manual jobs requires an environment declaration. Without one, any Developer can trigger the job. This is an important nuance for CI configuration.
Verify in UI:
  • Latest pipeline showing both manual jobs (deploy_prod gated, run_adhoc open): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/pipelines
  • Protected environments (prod allowed list = Carol only): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/ci_cd#js-protected-environments-settings

13.3 Scenario C — Fork Pipeline Isolation

Dan modified his fork to include the debug job from Scenario A and opened an MR. Fork pipelines were configured to require approval before running. Bob, acting as Trusted Committer, approved the pipeline.

When the fork pipeline ran, the debug job output showed DEV_CONFIG with its value but PROD_SECRET was blank. Even though the pipeline had been approved to run, the fork context did not grant access to protected variables.

Dan's attempt to navigate to service-a's CI/CD settings was rejected — he had no role on the upstream project and could not see variable keys or values.

Outcome: Fork pipelines were isolated from protected variables regardless of run approval. External contributors could not view variable configuration through the UI.
Verify in UI:
  • MR pipeline (run requires approval, debug job log shows PROD_SECRET blank): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/merge_requests?scope=all&state=all
  • Merge request settings — fork pipeline approval: https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/merge_requests
  • Dan's view of upstream CI/CD settings (should be blocked): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/ci_cd

13.4 Scenario D — CI_JOB_TOKEN Scoping

The default job token allowlist on service-a was inspected. It restricted inbound access via CI_JOB_TOKEN to the project itself.

A test job was added to service-b that attempted to clone service-a using the job token. The clone failed with a 403 authentication error. The allowlist on service-a was then explicitly updated to include service-b, and the clone succeeded on the next run. Removing service-b from the allowlist restored the deny behavior.

Outcome: Cross-project job token access required explicit allowlisting. The default was correctly restrictive.
Verify in UI:
  • Job token allowlist on service-a (target of inbound calls): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/settings/ci_cd#js-token-access
  • service-b pipeline (success/fail of clone job over time): https://<gitlab-host>/acme/verticals/domain-a/service-b/-/pipelines

13.5 Scenario E — Runner Scoping

A group-level runner was registered under acme/verticals/domain-a with the tag domain-a-only and the Protected flag enabled. A job in service-a tagged with domain-a-only ran successfully on main but remained in pending state when pushed to a feature branch, because the runner only picked up jobs from protected refs.

A second test project was created in acme/verticals/identity with an identical job tagged domain-a-only. The job remained pending indefinitely — the runner, being group-scoped to domain-a, did not see jobs from sibling verticals.

Outcome: Group-scoped runners correctly refused jobs from other groups. Protected runners correctly refused jobs from unprotected refs. Runner scoping provides a clean layer of CI isolation.

13.7 Scenario G — Scheduled Pipeline Ownership

Bob created a pipeline schedule on service-a running on main every hour. The schedule list showed Bob as the owner.

As a test, Bob's role on service-a was reduced from Maintainer to Developer. The schedule continued to fire, but jobs requiring Maintainer privileges (in this case, a deploy to prod) failed at runtime. Scheduled pipelines execute with the current permissions of their owner, not frozen-in permissions from the schedule's creation time.

Ownership of the schedule was then transferred via the "Take ownership" control to a long-lived account, removing the dependency on Bob's individual role.

Outcome: Scheduled pipelines inherit their owner's live permissions. Ownership must be managed deliberately, especially when the original owner changes role or leaves. Schedule ownership should be held by a service-style account for production-critical automation.
Verify in UI:
  • Pipeline schedules (owner column, Take ownership control): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/pipeline_schedules
  • Pipeline history filtered to scheduled runs (success/failure under reduced role): https://<gitlab-host>/acme/verticals/domain-a/service-a/-/pipelines?scope=all&source=schedule

13.8 Scenario H — Variable Inheritance and Override

A group-level variable SHARED_VAR was set at acme/verticals/domain-a. Jobs in both service-a and service-b were able to read it — confirming downward inheritance.

A project-level variable with the same key was then added to service-a with a different value. Jobs in service-a then read the project-level value, while service-b continued to read the group-level value. Project variables override group variables at the project scope.

A variable added to the sibling group acme/verticals/identity was not visible to jobs in service-a, confirming sibling isolation for variables.

Outcome: Variable inheritance followed the group hierarchy cleanly. Overrides behaved as expected. Sibling isolation held for CI variables just as it did for repository access.

15. Observations and Gaps

Several observations emerged during execution that are worth recording for the broader rollout:

  • Manual jobs without environment declarations fall back to role-level gating. Any CI pipeline that expects a specific user or team to trigger a deployment must declare an environment. This is a configuration convention that should be documented for all teams.
  • The visibility inheritance constraint (child cannot be more visible than parent) required planning when enabling inner source. Verticals intended to host inner-source projects must themselves be at Internal visibility, which affects discovery posture for the whole vertical.
  • Scheduled pipeline ownership is a long-lived attachment to an individual. In the absence of service accounts, schedules must have their ownership transferred whenever a Maintainer's role changes or they leave.
  • The default CI_JOB_TOKEN allowlist on newly created projects is appropriately restrictive. Older projects (pre-16.x) may need migration to the stricter defaults.
  • The Protect flag on variables and the environment scope on variables operate independently — both must be correctly set for true production-secret isolation. Documentation should emphasize this combination.
  • No integration with an identity provider was tested in this PoC. Manual user invitation was used throughout. The next PoC phase should validate that SAML Group Sync preserves these access behaviors when role assignments originate from IdP groups rather than direct invitation.

16. Recommendation

Go. All 25 success criteria passed. The GitLab access model, as designed, enforces the intended separation of duties across repository operations, CI/CD pipeline execution, and inner-source contribution. The model is ready for broader rollout.

Recommended next steps:

  1. Adopt the role model and hierarchy patterns validated in this PoC as the standard for all new verticals and projects.
  2. Plan a follow-up PoC to validate SSO / SCIM / SAML Group Sync on the same hierarchy, using IdP-driven membership instead of manual invitation.
  3. Plan a compliance framework PoC to add mandatory CI jobs (SAST, secret detection, dependency scanning) on top of the validated access model.
  4. Publish a short operations guide capturing the conventions discovered during this PoC (environment declarations on manual jobs, schedule ownership, variable configuration patterns, visibility inheritance).
