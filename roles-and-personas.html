Footnotes
  1. Bob can't push to main either. The protected-branch rule is "Allowed to push: No one; Allowed to merge: Maintainers". Bob can merge MRs into main but not push directly.
  2. Bob is a Maintainer but the prod environment's allowed-deployer list contains only Carol — being Maintainer is not enough to play the manual deploy job. This is the user-specific layer.
  3. Maintainer can add members up to Maintainer level. Adding an Owner requires an Owner.
  4. Carol plays the job, but the deployment then transitions to "waiting for approval" because the prod environment requires Bob's approval (two-person rule).
  5. Even if Carol were on the approver list, GitLab blocks self-approval on a deployment she initiated.
  6. Operator's admin_protected_environments ability covers environments. Carol does not have admin_protected_branches, so branch rules remain Maintainer-only.
  7. Sam approves via the admin_merge_request ability — designed for security-sensitive MRs (e.g. /infra/ per CODEOWNERS), without granting write access.
  8. Dan opens MRs from a fork (poc-dan/api), not the upstream. He can't create a feature branch on upstream domain-a/proj-1.
  9. Dan can run pipelines on his fork. When his fork-MR runs against upstream, fork pipelines require Maintainer approval to execute, and protected variables remain hidden.
  10. root can't push to main via the standard flow — protected-branch rule "Allowed to push: No one" applies even to admin. Admin can override by editing the protection rule first.
