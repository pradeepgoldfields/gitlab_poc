Enterprise GitLab Access Model PoC — Execution Guide

Purpose: Step-by-step instructions to execute the enterprise GitLab access model Proof of Concept. Follow top to bottom. Each step has a click path, values to enter, and a check to confirm it worked before moving on.

What you will build: A self-contained validation environment covering the deployment-model hierarchy, both authorization approaches (hybrid and target), the three custom roles, inner sourcing, confidential handling, zone-level policy inheritance, CI/CD access scenarios, and a migration between the two approaches.

Who this is for: One platform engineer executing the PoC manually through the GitLab web UI.

Time: 3–5 days of focused work. Can be spread over 2 weeks at 2 hours/day. The guide is organised in phases so you can pause and resume at phase boundaries.

Assumption: You have instance Admin rights on the self-managed GitLab instance, or at least Owner rights on a dedicated top-level sandbox group plus the ability to create instance-level Custom Roles (which requires Admin).

PHASE 1 — Custom Roles (Instance-Level Setup)

2. Phase 1 — Custom Roles

Custom Roles must be created at the instance level before any group can use them. This phase requires Admin.

Step 1.1 — Open Custom Roles admin

Navigate to Admin Area → Custom Roles (or /admin/custom_roles). Click New custom role.

Step 1.2 — Create Promoter role

Fill in:
  • Name: Promoter
  • Description: "Triggers promotion pipelines; no write access"
  • Base role: Reporter
Enable these abilities:
  • admin_cicd_variables
Do NOT enable: push_code, admin_project, merge_merge_request, admin_group_member.
Note (GitLab 18.x): The original design called for an admin_pipeline ability. That ability is not exposed by the Custom Roles API in 18.x — pipeline triggering is governed by base role + protected branch access. admin_cicd_variables on its own is sufficient for the Promoter use case (configure variables consumed by promotion jobs). The promotion trigger itself is handled by base Reporter + protected-branch allow-list.


Click Create custom role.
Check: The Custom Roles list shows "Promoter" with base Reporter and 1 custom ability.

Step 1.3 — Create Operator role

Click New custom role again. Fill in:
  • Name: Operator
  • Description: "Manages prod environment and release branches; restricted via protected environment allow-lists"
  • Base role: Reporter
Enable:
  • admin_protected_environments
  • admin_cicd_variables
  • read_runners
  • admin_runners
Do NOT enable: push_code, admin_project, merge_merge_request.
Note (GitLab 18.x): The original design named admin_pipeline + admin_environment. The 18.x API exposes admin_protected_environments (manage who can deploy where) and admin_runners (manage runner registration / pause). admin_runners requires read_runners as a prerequisite — the API rejects the create otherwise. Pipeline-trigger authority on prod still flows through the Operator's name being on the protected-environment allow-list (see Phase 7).

Click Create custom role.

Step 1.4 — Create Security Manager role

Click New custom role. Fill in:
  • Name: Security Manager
  • Description: "Manages vulnerability waivers and approves security-sensitive MRs"
  • Base role: Reporter
Enable:
  • read_vulnerability
  • admin_vulnerability
  • admin_merge_request
Do NOT enable: push_code, admin_project, merge_merge_request.
Note (GitLab 18.x): The original design named approve_merge_request. The 18.x API exposes admin_merge_request, which covers approval and reviewer management without granting the merge action itself (still gated by merge_merge_request, which we explicitly do not enable). Net effect for the Security Manager use case is the same: Sam can approve security-sensitive MRs (e.g., CODEOWNERS-required /infra/ paths) but cannot push or merge.


Click Create custom role.
Check: Admin Area → Custom Roles shows three roles: Promoter, Operator, Security Manager. Take a screenshot of this list for the final report.
PHASE 2 — Hierarchy Skeleton

Click Create group.

PHASE 7 — Branch, Tag, and Environment Protection

8. Phase 7 — Branch, Tag, and Environment Protection

Configure the protection rules that make roles meaningful. Apply to domain-a/proj-1 and domain-b/proj-1 as the two primary test projects.

Step 7.1 — Protect main on domain-a/proj-1

Navigate to domain-a/proj-1 → Settings → Repository → Protected branches. For main:
  • Allowed to merge: Maintainers
  • Allowed to push and merge: No one
  • Allowed to force push: No
  • Require approval from code owners: On

Step 7.2 — Configure approval rules

Navigate to Settings → Merge requests. Under Merge request approvals:
  • Prevent approval by author: On
  • Prevent approvals by users who add commits: On
  • Remove all approvals when commits are added: On
Add an approval rule:
  • Name: "Default"
  • Required approvals: 1
  • Approvers: acme-poc/iam-sim/p2p/IAM_DevOps_domain-a_Maintainer

Step 7.3 — Add CODEOWNERS

As poc-bob (Maintainer), add a CODEOWNERS file at the root of domain-a/proj-1: 
*                @acme-poc/iam-sim/p2p/IAM_DevOps_domain-a_Maintainer
/infra/          @acme-poc/iam-sim/p2p/IAM_DevOps_domain-a_SecurityManager
/.gitlab-ci.yml  @acme-poc/iam-sim/p2p/IAM_DevOps_domain-a_Maintainer
Commit directly to main.

Step 7.4 — Protect tags

Navigate to Settings → Repository → Protected tags. Add:
  • Tag pattern: v*
  • Allowed to create: Maintainers

Step 7.5 — Create protected environments

Navigate to Settings → CI/CD → Protected environments. Add: staging:
  • Allowed to deploy: Developers + Maintainers
prod:
  • Allowed to deploy: poc-carol directly (representing the Operator user list)
  • Required approvals: 1
  • Approvers: poc-bob (or the Maintainer group)
Note: the "prod" environment must be created before it can be protected. Create it via Operate → Environments → New environment first.

Step 7.6 — Add a minimal pipeline to domain-a/proj-1

As Bob, add .gitlab-ci.yml to domain-a/proj-1: 
stages:
  - build
  - deploy

build:
  stage: build
  script:
    - echo "Building..."

deploy_staging:
  stage: deploy
  script:
    - echo "Deploying to staging"
  environment:
    name: staging
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

deploy_prod:
  stage: deploy
  script:
    - echo "Deploying to PROD"
  environment:
    name: prod
  rules:
    - if: $CI_COMMIT_TAG
  when: manual
Commit to main.

Step 7.7 — Repeat on domain-b/proj-1

Repeat steps 7.1–7.6 on domain-b/proj-1, substituting the domain-b SailPoint groups in approval rules and CODEOWNERS. This gives you two fully protected projects — one under each approach.

PHASE 11 — Zone-Level Policy Inheritance

12. Phase 11 — Zone-Level Policy Inheritance

The design's headline benefit — enforce policy once at the zone level and have it inherit — is validated here.

Step 11.1 — Set a zone-level CI/CD variable

Navigate to acme-poc/live-production → Settings → CI/CD → Variables → Add variable.
  • Key: PROD_DEPLOY_TOKEN
  • Value: zone-secret-value-live-prod
  • Protect variable: On
  • Mask variable: On
  • Environment scope: prod

Step 11.2 — Register a zone-scoped runner

Navigate to live-production → Build → Runners → New group runner.
  • Tag: live-prod-runner
  • Run untagged jobs: Off
  • Protected: On
Register it on any available host (a Docker container works for the PoC).

Step 11.3 — Verify inheritance to domain-a/proj-1 and domain-b/proj-1

Add a debug job to domain-a/proj-1 .gitlab-ci.yml: 
debug_prod_token:
  stage: build
  tags:
    - live-prod-runner
  script:
    - echo "TOKEN is ${PROD_DEPLOY_TOKEN:-empty}"
  environment:
    name: prod
  rules:
    - if: $CI_COMMIT_TAG
Push a v* tag.
Expected: The job picks up on the live-prod-runner and the token value is present (masked in logs, but non-empty).

Step 11.4 — Verify non-live does NOT inherit

Create a project non-live-enterprise/domain-a/test-service. Add the same debug job.
Expected: The job does NOT pick up on live-prod-runner (it is group-scoped to live-production and will remain pending). The PROD_DEPLOY_TOKEN is empty in the log. Confirms zone isolation.

Step 11.5 — Add a new project mid-test

Create live-production/domain-a/reporting. Add the same debug job.
Expected: The new project immediately has access to the zone runner and variable with no additional configuration. This is the core inheritance proof.

Common Pitfalls

  • Custom Roles not appearing in role dropdown. Wait a minute after creating; the cache may need to refresh. If still missing, confirm the user creating the membership has permission to assign custom roles (Maintainer or higher on the group).
  • Group sharing does not grant expected access. GitLab group sharing requires both: the sharing to be configured AND the recipient user to be a member of the shared group. Double-check both sides.
  • Restricted projects visible to parent domain users. The restricted subgroup itself must be Private — not merely its projects.
  • Protected variable visible on feature branch. Both the variable must be "Protected" AND the branch must be protected. Either missing and the variable is exposed.
  • Environment-scoped variable not visible. The job must declare environment: <name> matching the variable's scope exactly. Wildcards match anything; specific scopes require exact match.
  • Manual job playable by anyone with Developer role. Without an environment: declaration, only role gates apply. Add environment protection to restrict to specific users.
  • Scheduled pipeline fails after role change. Schedules run as their owner's current role. Transfer ownership to a long-lived account before role changes.
  • CI_JOB_TOKEN unexpectedly allows cross-project. On older GitLab versions the default was permissive. Check and tighten the allowlist per project.
  • Fork pipeline does not run. Either fork pipelines are disabled or require approval. Check Settings → Merge requests → Fork MR pipelines.
  • Session caching after role change. GitLab can cache permissions per session. After changing a user's role or group membership, have them log out and back in.
