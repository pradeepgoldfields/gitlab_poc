Enterprise GitLab Access Model PoC v2 — Comprehensive Overview

Single-document briefing for the v2 PoC: org-driven access with a three-tier separation between IAM source (LDAP), organisational hierarchy, and application hierarchy. v1 (single-tier, IAM→app direct) is preserved intact at gitlab-poc-scripts/ and at the acme-poc top group on the live instance for comparison.

What's different in v2. v1 grants access to applications by sharing IAM-sim subgroups directly with the application projects. v2 introduces a middle tier — the organisations/ tree — through which all access must flow. Apps own no direct shares; org subgroups own all the privilege; IAM groups populate the org subgroups. The lifecycle benefit: when a person moves teams in IAM, they automatically lose access to the old apps and gain access to the new ones. Zero changes at the application tree.
Important caveat — group-sharing transitivity. The v2 design is structurally clean and the live verifier passes 28 of its 29 checks, but the one failing check is a real-world limitation of GitLab's runtime permission resolver. Read Section 7 — Group Sharing in GitLab for the full explanation. Spoiler: the model works correctly under real LDAP/SAML sync (which the PoC simulates but cannot fully reproduce on a manual local install).

2. The v2 hierarchy

The v2 PoC has the same four top-level peer subgroups as v1 (iam-sim, platform, plus the new applications and organisations trees), but the meaning of "where access lives" changes fundamentally.

Figure 1 — v2 hierarchy. The new organisations/ tree (blue) sits as a peer of applications/. Every grant onto an app flows from an org subgroup, never directly from iam-sim.

3. The two-hop access flow

Hop 1 — IAM-sim → Organisation tree

Each LDAP group in iam-sim/sailpoint/ or iam-sim/sscam/ is bound to exactly one role-bearing org subgroup. The binding is recorded in config.IAM_TO_ORG_SHARES.

In production with real LDAP/SAML sync, GitLab's Group → LDAP synchronization setting on each org subgroup pulls members from the bound LDAP group on every sync interval and writes them in as direct members of the org subgroup. The LDAP group itself doesn't appear in GitLab — only the membership does.

The PoC simulates this by having Phase 4 read both the IAM_MEMBERSHIPS map (who is in which LDAP group, for the audit trail) and IAM_TO_ORG_SHARES, and provisioning each user as a direct member of the matching org subgroup. The IAM-sim subgroup itself is also populated so the "who is in which LDAP group" view in the UI is faithful.

Dual-feed example. Bob is in both sailpoint/gl-squad-1-squad-leads AND sscam/proj-1_w — modelling the partial-migration state where some apps are still on the legacy SSCAM naming. Both feeds populate the same target subgroup, tribe-1/squad-1/squad-leads; Bob ends up there exactly once.

Hop 2 — Organisation → Application

Each role-bearing org subgroup is shared with exactly one application scope at a specific role. The bindings live in config.ORG_TO_APP_SHARES. Most shares are at the subdomain group level (Hop 2 inherits to all projects beneath); a few are at the project level for finer slicing (e.g. squad-2/developers shares only with proj-3).

Built-in roles (Developer, Maintainer, Owner) are passed via the group_access parameter; custom roles (Promoter, Operator, Security Manager) are passed via member_role_id in the same call.

7. Group Sharing in GitLab — design, behaviour, and the transitivity caveat

GitLab's group-sharing feature (group → group and group → project) is the foundation of the v2 model. Understanding exactly how it behaves is essential to understanding why the PoC's end-to-end check fails on a manually-built instance and works in production.

What "share group A with group B" actually does

When you go to group B's Members → Invite a group and pick group A at role R, GitLab records a group link from B to A. After that:

  • Every direct member of A appears on B's Members list with role min(their role in A, R), labelled "via shared group A".
  • The membership flows down the hierarchy: every project in B inherits those members at the same role, just as if they had been added directly to B.
  • A user looking at "their groups" or "their projects" sees B and its contents listed.

Where the transitivity gap appears

Now consider a chain: X is shared with A, and A is shared with B. Is a member of X effectively a member of B?

The answer in GitLab today is "yes for the membership-listing endpoints, but no for the runtime permission resolver":

  • GET /groups/B/members/all walks the share chain and lists X's members with their effective access level.
  • GET /projects/<B-child>/members/all also lists them.
  • BUT: when an X member's HTTP request hits a B-child resource, GitLab's authorization code does not walk transitively through chained group shares. The user is treated as not-a-member, and the request returns 404 (Private project) or 403 (Internal/Public project, action requires a role).

This is the failing check. v2 deliberately keeps two separate trees (applications/ and organisations/) so the design intent — apps own no access, orgs own all access — is visible in the hierarchy itself. Hop 2 is a group share from each org subgroup to the corresponding app namespace. Inside the org subgroup, members are direct (LDAP-sync simulation). But the join from "member of org subgroup" to "member of app project" relies on GitLab walking the share chain at request time — which it doesn't.

Why this works in real production

Real LDAP/SAML-bound GitLab uses one of two patterns that sidestep the transitivity gap:

  1. Direct LDAP sync into the app tree. The org subgroup is moved into (or recreated as) a subgroup of the app tree — e.g. applications/domain-a/subdomain-a/squad-1-developers. LDAP sync writes squad members directly into this subgroup. Inheritance is then a straight downward walk through the namespace tree, which GitLab evaluates correctly. The org subgroup name still encodes the org concept; it just lives inside the app tree.
  2. LDAP-bound app-level shares. The LDAP group is bound directly to the application namespace. The org tree exists only as a documentation/UI structure (e.g. labels) and does not participate in access resolution.

Both patterns produce correct runtime evaluation, but pattern (1) preserves the visual separation v2 demonstrates. It's the recommended production migration path.

Why the PoC keeps the trees structurally distinct anyway

The whole point of v2 is to make the org-driven model visible in the hierarchy. Collapsing the org tree into the app tree (production pattern 1 above) optimises for runtime evaluation but hides the design intent — a reader looking at applications/domain-a/subdomain-a/squad-1-developers can't easily tell that this is an "org" thing rather than an "app" thing.

So the PoC keeps them apart, accepts that one specific runtime check fails on the local manually-built instance, documents why (this section), and recommends the production migration path (LDAP-sync-to-app-subgroup) so the runtime check would pass once real LDAP is wired up.

The verifier marks this honestly. The "End-to-end: Alice can GET proj-1 via her own PAT" check is the only one in the v2 verifier that exercises GitLab's runtime permission resolver (it mints a real PAT for Alice and makes a real GET request). On the local PoC instance, it returns HTTP 403 / 404. The check's detail field on the report explains the cause and the fix — no hand-waving, no marking the failure as a "false positive".
