Bitbucket to GitLab Migration — Analysis

Purpose: This analysis defines the migration strategy for moving from the existing Bitbucket platform to GitLab, aligned with the reference access model. It covers the target-state shape, the phased execution plan, and the concrete workstreams required to move repositories, history, access, CI/CD, integrations, and operational practices without disrupting delivery.

Audience: DevOps Tooling platform engineering, DevOps leadership, application teams preparing for onboarding, IAM owners, and release managers responsible for the cutover.

Scope: End-to-end migration of source code, branches, tags, pull request history (where preserved), Bitbucket Pipelines / Jenkins jobs, Bitbucket access groups (_r / _w), webhooks, and downstream integrations. Two authorization approaches (hybrid transition → target group-centric) as defined in the reference design are used as the anchoring model — this analysis adds the how on top of the what.

Out of scope: Instance sizing / hardware procurement, SSO federation design (assumed provided), the broader Path-to-Production redesign, Jira / Confluence migration (handled separately).

1. Migration Principles

The migration is governed by the same core principles as the target access model (reference design), extended with migration-specific guardrails:

  • No big-bang cutover. Applications migrate in waves. The two Bitbucket and GitLab platforms coexist during the transition.
  • Preserve the audit chain. Git history (commits, branches, tags) is migrated verbatim. Pull request discussions are preserved where technically feasible; otherwise a read-only Bitbucket archive is kept accessible.
  • Start with Approach 1, converge to Approach 2. On day one of an application's onboarding, reuse the Bitbucket _r / _w groups via SSCAM to avoid access-model changes colliding with the platform change. Move to the group-centric target model after the code has stabilised on GitLab.
  • Freeze writes during cutover. Each application enters a short write-freeze on Bitbucket during the final sync. No merges, no tag pushes. Duration measured in hours, not days.
  • Inner sourcing from day one. Non-confidential projects are Internal in GitLab from the moment they land. Confidential projects go into restricted/ subgroups before the first push.
  • CI/CD is re-platformed, not lifted. Bitbucket Pipelines and Jenkins configurations are rewritten to .gitlab-ci.yml. A direct copy is usually wrong — use the migration as an opportunity to apply the enterprise pipeline templates.
  • Rollback is always available until decommissioning. Bitbucket remains readable and restorable per project until the project has been verified green on GitLab for the agreed soak period.

Common trap: Teams often assume pull-request history migrates automatically. The built-in GitLab importer for Bitbucket covers code, branches, and tags, plus a best-effort import of PRs as MRs — but comments, approvals, and the reviewer graph are not always preserved. Plan for a read-only Bitbucket archive if PR discussions are a compliance artifact.

4. Migration Phases (High-Level Plan)

The migration runs in four phases. Phases 0 and 1 are one-time platform work. Phases 2 and 3 repeat per wave.

Figure 2 — The four migration phases. Phase 0 and Phase 1 are one-time. Phase 2 repeats per wave. Phase 3 begins per domain once Phase 2 is stable for that domain.

5. Workstream A — Repository & History Migration

Two viable paths. Pick per repo based on how much metadata must be preserved.

5.1 Path A1 — GitLab's built-in Bitbucket importer

Uses OAuth from GitLab to Bitbucket Cloud / Server. Imports code, branches, tags, LFS, and a best-effort version of pull requests. Recommended when:

  • Repos are not large (< 2–3 GB).
  • Some loss in PR metadata fidelity is acceptable (comments may be preserved as MR notes, but reviewer graph and approvals are not fully reproduced).
  • The source is reachable from the GitLab instance.

5.2 Path A2 — Mirror via bare clone

The controlled path. Clone with --mirror, push to an empty GitLab project. Gives exact history fidelity but does not carry PRs at all. Use when:

  • Repos are large or contain LFS that must be verified.
  • PR history is being archived in a read-only Bitbucket snapshot (acceptable for compliance in most cases, since the Bitbucket system-of-record remains queryable).
  • Automation over 100s of repos is required.
# Per-repo migration script (sketch)
git clone --mirror https://bitbucket.example.com/scm/PAY/domain-a-api.git
cd domain-a-api.git
git remote set-url --push origin https://gitlab.example.com/acme/domain-a/proj-1.git
git push --mirror
# Then verify: branch + tag counts, LFS object counts, sample-clone checks

5.3 Pull request preservation

Three options, choose one per application and document it:

  1. Best-effort import — use the built-in importer and accept the fidelity loss.
  2. Read-only Bitbucket archive — keep the old Bitbucket project in read-only mode for N months. Link from the GitLab project's README.
  3. Cold archive — export PR data as JSON via the Bitbucket API and store it in an object store. Useful when Bitbucket is fully decommissioned.
Recommended default: Path A2 + Option 2. Use the mirror-clone for deterministic code migration and keep Bitbucket read-only for the duration of the wave's soak period. This separates "code is on GitLab" from "historical PR search still works".

6. Workstream B — Access & LDAP Migration

This workstream follows the reference design directly. The analysis below adds the sequencing.

Figure 3 — Access migration sequencing per application. Steps 2 and 3 keep the Bitbucket-era groups working so users see zero access disruption. Step 4 is only entered once the new groups are proven, at which point the legacy _w / _r groups are decommissioned.

6.1 LDAP Group Sync configuration

GitLab binds LDAP groups to GitLab groups through Group Sync. The bindings are configured per GitLab group; the LDAP side is managed by the existing SSCAM / SailPoint systems.

  • For Approach 1 onboarding, bind the existing <app>_w / <app>_r LDAP groups to the target GitLab project at Developer and Reporter respectively.
  • For Approach 2 convergence, bind IAM_DevOps_<group>_Developer / _Reporter / _Maintainer / _Owner LDAP groups to the target GitLab group (domain level) at the corresponding role.
  • Custom roles (Promoter, Operator, Security Manager) bind IAM_DevOps_<group>_Promoter / _Operator / _SecurityManager at group scope.
  • Audit: export the membership of each newly-bound group before cutover; verify post-cutover that the set of resolved GitLab memberships matches.
Watch sync latency. LDAP Group Sync runs on an interval (typically hourly). Do not start write-freeze cutover until a full sync cycle has completed and memberships have been verified.

7. Workstream C — CI/CD Pipeline Migration

This workstream concentrates the most effort. Bitbucket Pipelines and Jenkins configurations do not auto-convert to .gitlab-ci.yml. The migration is also the right moment to apply enterprise pipeline templates, eliminating drift accumulated over years.

7.1 Conversion strategy

SourceTargetApproach
Bitbucket Pipelines (bitbucket-pipelines.yml).gitlab-ci.yml using enterprise templates via include:Manual rewrite per repo, guided by a template catalogue. Parallelisable across teams.
Jenkins multibranch pipelines (simple, "just build + test")GitLab CIRewrite to .gitlab-ci.yml. Decommission Jenkins job.
Jenkins jobs with heavy plugin dependencies (e.g. blue-green orchestration, custom slaves)Keep Jenkins, trigger via GitLab CI webhookBridge solution. Add a Jenkins-trigger job to the pipeline until Jenkins work is rewritten natively.
Shell scripts & Makefiles used by CIUnchanged — invoked from .gitlab-ci.ymlNo change; reuse as-is.
Secrets in Bitbucket Pipelines repository variablesGitLab CI/CD variables, masked + protected, scoped to the group where possibleRotate on the way across. Never transcribe — create fresh.
Artifact publishing to Nexus / ArtifactorySame target, new credentialsReissue publish credentials; verify upload from a GitLab runner before cutover.

7.2 Runner topology

  • Shared runners — for general build/test of internal-only projects. Cheapest to operate.
  • Group-scoped runners — for projects needing access to deployment-zone-specific networks (e.g. a runner in the cloud landing zone, with egress to the relevant VPC).
  • Project-scoped runners — exceptional. Only for strict tenancy requirements.
  • Tag runners with capability labels (e.g. zone:live-prod, os:rhel8). Pipelines declare required tags.

7.3 Dual-running CI during cutover

For the window where both Bitbucket and GitLab are accepting code (wave cutover window), disable CI on the Bitbucket side once the GitLab pipeline is green on the last synced commit. Avoid running builds on both platforms simultaneously — deploys get confusing, and you burn minutes twice.

8. Workstream D — Integrations, Webhooks, and Tooling

Every external system that knows about the Bitbucket URL or API needs to know about GitLab instead. Cataloguing these early is the highest-leverage work of the foundation phase, because each one is a potential cutover blocker.

Don't forget shell scripts. Internal scripts, CI bootstrap code, and documentation often hard-code Bitbucket URLs. Grep the enterprise codebase for bitbucket.example.com before the cutover window. A broken bootstrap script discovered at 2am is the single most common migration incident.

9. Cutover & Freeze Strategy per Application

Each application follows the same cutover template during Phase 2. The window is small (hours to a single day), executed by the platform team with the application owner present.

Figure 4 — Cutover timeline per application. The write-freeze is bounded (typically 1 hour) and the soak period keeps Bitbucket readable so the team has a rollback anchor.

9.1 Cutover checklist (per app)

  1. T-7d: create the GitLab project under the correct deployment zone / domain; configure Approach 1 access; mirror the repo; configure the runner tags; translate CI to .gitlab-ci.yml.
  2. T-3d: run the GitLab CI against the latest Bitbucket main; verify artefacts match; register webhooks.
  3. T-1d: announce freeze window; stop scheduled pipelines on Bitbucket.
  4. T-1h: enter write-freeze on Bitbucket (permissions → read-only for all, or a branch-level block on main and tag pushes).
  5. T0: final mirror sync (git push --mirror); verify last-commit SHA matches on both sides.
  6. T0+30m: switch external webhooks (Jira, Sonar, Slack) from Bitbucket to GitLab.
  7. T0+1h: enable GitLab CI; trigger a canary build.
  8. T0+2h: hand the app back to the team; Bitbucket remains read-only.
  9. T0+14d (soak end): confirm no fallback needed; decommission the old Bitbucket repo (archive flag, not deletion).

10. Rollback Strategy

A rollback is the act of reverting an application's system-of-record from GitLab back to Bitbucket. It should be a rare event, but must remain possible during the soak period.

SituationAction
Cutover sync failed mid-way (T0 window)Stay on Bitbucket. Lift the freeze. Re-run the migration after root-cause analysis.
Critical CI/CD issue surfaces within 24hKeep code on GitLab. Fall back to Jenkins/Bitbucket Pipelines as a stopgap via a webhook from GitLab. Fix GitLab CI and retire the stopgap within the soak window.
Unrecoverable operational issue within soak period (rare)Re-open Bitbucket for writes for that project, re-mirror any commits made on GitLab back to Bitbucket (git push bitbucket +refs/heads/*:refs/heads/* from a clone of GitLab), freeze GitLab, fix, re-cutover.
Issue discovered after soak endNo rollback path. Treat as a forward-fix problem on GitLab. The Bitbucket archive remains read-only as a reference.
The escape valve: the only rollback window that matters is the soak period. Communicate it explicitly to application owners: "For 14 days after cutover you can be reverted to Bitbucket if something is fundamentally broken. After that, we forward-fix on GitLab."

12. Success Criteria

Migration of an application is considered successful when all of the following hold at the end of its soak period:

  • Default branch tip on GitLab matches the Bitbucket tip at the final sync (commit SHA).
  • All branches and all tags are present on GitLab with matching SHAs.
  • LFS object count on GitLab matches Bitbucket.
  • The production build can be reproduced from the GitLab repo using the new pipeline, and the artefact matches the last Bitbucket-produced artefact by digest (or by documented equivalent if the build is non-deterministic).
  • All authorised users have access to the GitLab project via either the legacy _w/_r binding or the new group-level binding — no-one is locked out.
  • External integrations (Jira, Sonar, Slack, deploy) are operating against GitLab URLs.
  • Bitbucket repo is read-only and still readable for history queries.
  • No outstanding P1 defects against the migration for 14 consecutive days.

Migration of the platform is successful when:

  • All applications in scope have cleared their per-app soak.
  • Each domain has either converged to Approach 2 or made an explicit, documented decision to remain on Approach 1.
  • Bitbucket is decommissioned (or placed into cold archive per the retention policy).
  • An access review cycle has run end-to-end through GitLab + SailPoint.

13. Recommended Wave Sequencing

Wave composition matters as much as wave size. A wave made entirely of one domain's apps lets you converge to Approach 2 for that domain at the end of the wave. A wave of unrelated apps spreads risk but leaves each domain half-migrated.

Figure 5 — Suggested wave sequencing. Pilot and Wave 1 prove the model in low-risk territory. Waves 2 and 3 handle the bulk, sorted by deployment zone. Convergence to Approach 2 happens per domain on its own schedule, not on the migration's schedule.
Why deployment-zone-ordered, not team-ordered: the deployment model is the design's primary criterion for grouping decisions, even though it is not expressed as group nesting in the GitLab tree. Migrating zone-by-zone means the policy templates, protected-environment rules, and runner registrations for that zone can be refined and applied wholesale. It also means the highest-risk zone (live-production deployments) goes last, with the most learning accumulated. The zone is captured by the environment names, scoped variables, and runner tags — not by the path in the group tree.

Appendix — Consistency with the Reference Design

This analysis is consistent with the reference access model in [analysis](analysis):

  • Approach 1 (Hybrid Transition) is the onboarding model for every application during Phase 2.
  • Approach 2 (Domain / Product-Line Group-Centric) is the target state and the entry criterion of Phase 3 per domain.
  • Deployment-model criterion (live-production, non-live, etc.) drives wave sequencing, environment names, scoped CI variables, and runner topology — but is not expressed via deployment-zone subgroups in the tree (those would force per-zone duplication of every multi-environment app).
  • Inner sourcing by default is enabled from the first cutover, via Internal visibility on non-restricted projects.
  • Confidential projects are placed into restricted/ subgroups before any push.
  • The custom roles (Promoter, Operator, Security Manager) are provisioned once in Phase 0 and bound per domain during Phase 3.

If the reference design changes (role naming, grouping criterion, approach definitions), this analysis must be re-read for consistency and updated.
